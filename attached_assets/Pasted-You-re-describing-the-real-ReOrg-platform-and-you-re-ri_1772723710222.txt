# What the platform must become (non-negotiable architecture)

## 1) True multi-tenant consultant platform

* **Tenant = consulting firm**
* Tenant contains users + RBAC + policies + billing
* Tenant creates **Companies (clients)**
* Each company has **Projects**
* Each project has **Phases 0–7** with **gated progression** and versioned outputs

## 2) Phase engine + versioned artifacts (this is the backbone)

Every phase needs:

* `phase_run` (status, logs, inputs, outputs summary)
* versioned `artifacts` (JSON source of truth)
* `gate` decision record (pass/fail, who approved, why, evidence coverage)

This is how you get repeatable, “consulting-grade” outputs.

## 3) Dashboards and graphs must come from a metrics layer (not LLM prose)

To produce sophisticated dashboards like your examples (KPI dashboards, ROI/NPV, stack cost models, risk heatmaps), you need:

* canonical metrics tables (`metrics_timeseries`, marts/materialized views)
* consistent chart pipelines
* “show data” + “show evidence” on every chart/number

LLM writes the narrative; **the platform computes the numbers**.

## 4) Docling-like normalization for massive mixed files

Your ingestion must produce **Document Bundles** so the LLM can reliably work at scale:

* `manifest.json` (pages/tables/anchors)
* `text.md` (clean text)
* `chunks.jsonl` (chunked text with anchors)
* `tables/*.json` (table extraction with cell anchors)
* stored in object storage, indexed for retrieval

This is how you handle *massive* PDFs/XLSX/DOCX/PPTX without losing traceability.

# “LLM must be very precise and tuned”

You won’t get enterprise precision by asking a single model to “generate the report.” You get it by enforcing a quality system:

## The enterprise-grade LLM setup

* **Structured outputs only** (JSON validated against schemas)
* **Evidence-first**: every claim links to anchors (Evidence Drawer)
* **Deterministic numbers**: financial/KPI math runs in numeric blocks, not in the LLM
* **Multi-agent workflow**:

  * Classifier (low risk)
  * Extractor (facts+evidence)
  * Verifier (evidence coverage + unit/period checks)
  * Narrator (prose only from approved facts)
  * Editor (enterprise tone + template conformance)
* **Evaluation harness** in CI:

  * schema pass rate
  * evidence coverage %
  * “orphan number” detection in narrative
  * regression suite on sample bundles

Fine-tuning can come later—after you have stable schemas, metrics, and gating. Otherwise you’ll just train instability.
